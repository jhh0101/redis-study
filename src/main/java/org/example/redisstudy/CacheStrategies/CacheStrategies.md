## 1. Look-Aside (지연 로딩, Cache Aside) - 조회 최적화
데이터를 조회할 때 캐시를 먼저 확인하고, 캐시에 데이터가 없을 때만 DB에서 데이터를 읽어오는 가장 대중적인 패턴입니다.

### 작동 흐름
* 애플리케이션이 Redis(캐시)에 데이터를 요청합니다.  
* 데이터가 있으면(Cache Hit) 바로 반환합니다.
* 데이터가 없으면(Cache Miss) DB에서 데이터를 조회합니다.
* 조회한 데이터를 Redis에 저장하고 클라이언트에게 반환합니다.

### 장점
* Redis가 다운되더라도 DB를 통해 서비스가 계속 동작할 수 있습니다(장애에 강함).
* 실제로 자주 요청되는 데이터만 캐시에 저장되므로 리소스 낭비가 적습니다.

### 단점
* 초기 부하 (Cold Start): 캐시가 비어있는 초기 상태에서는 모든 요청이 DB로 몰려 부하가 발생할 수 있습니다.
* 데이터 정합성 문제: DB의 데이터가 수정되었는데 캐시가 갱신되지 않으면, 사용자는 과거의 데이터를 보게 됩니다.
### 적용 예시 (상품 상세 조회)
경매 상품의 상세 정보나 이미지 URL 등은 읽기 요청은 매우 많지만 데이터가 자주 변경되지는 않습니다. 이때 Look-Aside를 적용하면 DB의 조회(Read) 부하를 극적으로 줄일 수 있습니다.

## 2. Write-Through (동시 쓰기) - 조회 최적화(Redis와 DB의 정합성이 중요할 때 사용)
데이터를 저장할 때, 무조건 캐시와 DB에 동시에 업데이트하는 패턴입니다.

### 작동 흐름
* 애플리케이션이 데이터를 쓸 때, 먼저 Redis에 데이터를 저장합니다.
* 곧바로 이어서 DB에도 같은 데이터를 저장합니다.

### 장점
* 캐시와 DB의 데이터가 항상 100% 일치합니다(강한 데이터 정합성).
* 데이터 유실 위험이 없습니다.

### 단점

* 모든 쓰기 작업마다 두 번의 저장(Redis + DB)이 발생하므로 쓰기 지연 시간(Latency)이 증가합니다.
* 잘 읽지 않는 데이터까지 모두 캐시에 저장되므로 메모리 낭비가 심합니다 (보통 TTL 설정으로 보완합니다).

## 3. Write-Behind (Write-Back, 모아서 쓰기) - 쓰기 최적화
데이터를 캐시에만 먼저 빠르게 써두고, 나중에 특정 주기나 조건에 따라 모아서(Batch) 한 번에 DB에 반영하는 패턴입니다.

### 작동 흐름
* 애플리케이션의 모든 쓰기 요청을 Redis에만 저장하고 클라이언트에게 성공 응답을 보냅니다.
* 별도의 스케줄러나 워커가 Redis에 쌓인 데이터를 비동기적으로 DB에 일괄 저장(Sync)합니다.

### 장점
* 디스크 I/O가 발생하는 DB 쓰기를 건너뛰기 때문에 쓰기(Write) 성능이 압도적으로 빠릅니다.
* DB에 가해지는 트래픽 부하를 엄청나게 줄일 수 있습니다.

### 단점
* DB에 데이터를 동기화하기 전에 Redis 서버가 다운되면 데이터가 영구적으로 유실될 위험이 있습니다.
* 구현이 복잡하고, 데이터 유실을 막기 위한 추가적인 아키텍처(예: Redis AOF/RDB 설정, 메시지 큐 등)가 필요할 수 있습니다. 

### 적용 예시 (경매 입찰 로직)
마감 직전 수백 명의 사용자가 동시에 입찰 버튼을 누르는 상황을 상상해 보세요. 이 트래픽을 모두 DB의 JPA 비관적 락 등으로 처리하면 병목 현상이 발생합니다. 이때 Write-Behind를 적용해 입찰가 갱신을 Redis의 메모리상에서만 초고속으로 처리하고, 최신 입찰 상태만 주기적으로 DB에 스냅샷 뜨듯 저장하면 트래픽을 완벽하게 방어할 수 있습니다.
