## 왜 유령 키(Phantom Key)가 생길까?
레디스 자체의 만료 기능은 만료되는 순간 데이터를 아예 날려버리기 때문에, 스프링 입장에서는 "누가 죽었는지" 원본 데이터를 확인할 길이 없음

### 그래서 스프링은 이렇게 사용함.   
* 원본 데이터: refreshToken:user123 (TTL: 무제한으로 설정)
* 유령 데이터: refreshToken:user123:phantom (TTL: 14일 설정)
* 스프링이 유령 데이터가 14일 뒤에 소멸하는 걸 감지함
* 소멸을 감지하면 그제야 원본 데이터를 꺼내서 이벤트를 발생시키고, 원본 데이터도 같이 삭제함
**결론**: 편의성(이벤트 처리)을 위해 데이터가 2배로 저장되는 오버헤드가 발생한다. 만약 만료 이벤트 처리가 전혀 필요 없고 그냥 지워지기만 하면 된다면, 
설정(@EnableRedisRepositories(enableKeyspaceEvents = RedisKeyValueAdapter.EnableKeyspaceEvents.ON_DEMAND))을 조절하거나 원시적인 RedisTemplate을 쓰는 것이 낫다.
>@EnableRedisRepositories
>* 의미: "이제부터 JPA 쓰듯이 레디스 리포지토리(CrudRepository) 기능을 켜겠다!"라는 메인 스위치
>* 보통 RedisConfig 같은 설정 클래스 위에 붙임

>enableKeyspaceEvents = ...
>* 의미: "레디스 안에서 벌어지는 이벤트(특히 데이터 만료/삭제)를 스프링이 실시간으로 엿들을지 말지"를 정하는 옵션
>* 레디스에는 Keyspace Notifications라는 기능이 있는데, 데이터가 지워질 때 Pub/Sub 방식으로 "누구누구 데이터 방금 지워짐!" 하고 방송을 해줌. 스프링은 이 방송을 구독할지 말지 정함.

>ON_DEMAND (이게 핵심!)
>* 3가지 옵션 중 ON_DEMAND는 가장 똑똑하고 효율적인 방식
>* OFF (기본값): 알림 방송 무시. (데이터가 만료돼서 사라져도 스프링은 모름. 유령 키도 안 생겨서 메모리는 가장 아낄 수 있음)
>* ON_STARTUP: 스프링 서버가 켜지자마자 무지성으로 "레디스야, 앞으로 지워지는 데이터 있으면 무조건 다 나한테 알려줘!" 하고 구독해 버림
>* ON_DEMAND (수요에 따라): 스프링이 우리가 짠 코드를 싹 훑어봄. 그리고 @TimeToLive가 붙어있는 엔티티(예: RefreshToken)를 발견했을 때 알림을 구독함

>@EventListener으로 이벤트를 감지해서 서비스 로직을 실행(서비스 메서드에서 사용)

## 모든 기술은 트레이드오프(Trade-off)

>@RedisHash를 적극 추천하는 경우
>* 리프레시 토큰, 이메일 인증 코드, 장바구니처럼 구조가 있는 객체를 며칠~몇 주 정도 저장해야 할 때.
>* 빠른 개발 속도가 생명이고, 레디스 메모리가 넉넉할 때.

>RedisTemplate으로 돌아가야 하는 경우:
>* 데이터가 수백만 건 단위로 쌓여서 메모리를 극한으로 아껴야 할 때 (유령 키와 인덱스 키가 부담됨).
>* 단순한 숫자(조회수, 카운터)나 단순한 문자열만 저장할 때.
